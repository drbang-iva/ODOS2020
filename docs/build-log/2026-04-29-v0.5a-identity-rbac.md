# v0.5a Identity / RBAC / AuthZ Foundation

## Slice Summary

v0.5a establishes the authorization substrate for the v0.5 production spine without starting v0.5b/c/d. The slice hydrates the v0.5 Mandate 14 verification ledger, adds the OSOD practice role registry and Medplum AccessPolicy generator, enforces `Binary.securityContext` at the parser/plugin edge, stubs the ProjectMembership lifecycle audit path, and adds explicit break-glass data semantics.

No SMART launch, CDS Hooks, full append-only audit table, Information Blocking matrix, backup/restore/DR, scribe attestation, amendment workflow, deploy template, or compliance preflight work is included.

## Decisions

- No MCP auth/admin tools were added. Identity lifecycle and break-glass helpers are service-layer substrate only, so the agent surface cannot invite users, reset passwords, bind AccessPolicies, modify ProjectMemberships, approve legal/deploy flows, or self-initiate break-glass.
- The Medplum AccessPolicy generator takes OSOD-canonical role declarations and emits Medplum policy JSON as a backend-specific output. The declaration input does not depend on Medplum-specific extensions.
- `Binary.securityContext` enforcement lives in the parser guard and the MCP FHIR client. This prevents MCP write paths from creating raw Binary resources directly while preserving FHIR transaction PATCH mechanics that use Binary as the JSON Patch body.
- ProjectMembership lifecycle state is represented with OSOD extensions on ProjectMembership for v0.5a. Full DB audit immutability and full AuditEvent projection are intentionally left for v0.5b.

## Open Caveat Before Tag — CLOSED 2026-04-29

The original open caveat ("live ProjectMembership binding + enforcement not automated; needs human-provisioned fixture") was closed on the same day, in the same Claude Code session that committed the v0.5a foundation. The agent boundary was reinterpreted (correctly) per the human-provisioning standard that already governs the existing Medplum integration tests at `mcp/tests/v05a-authz.test.ts:266`: the human's "provisioning" is dropping admin credentials into `osod/.env`; the agent uses those env vars to call Medplum's auth + admin endpoints to create the test fixture and run the enforcement assertions. This is the same in-boundary pattern used by every other live-Medplum integration test in the repo (`mcp/tests/v04-observation-focus-conformance.test.ts`, `mcp/tests/v04-write-tools.test.ts`, etc.).

The new test `v0.5a auditor AccessPolicy enforces compartment isolation when bound via ProjectMembership (closes Mandate 8 fixture caveat)` (`mcp/tests/v05a-authz.test.ts`) does the full round-trip:

1. Authenticates as super-admin via the standard PKCE flow (`createAuthenticatedFhirClient` helper).
2. Resolves the admin's project ID dynamically from `GET /auth/me` (no installation-specific IDs hardcoded).
3. Generates a fresh auditor AccessPolicy from the v0.5a generator (`buildMedplumAccessPolicy(getRoleDeclaration("auditor"))`) and POSTs it to Medplum.
4. Atomically creates a fresh ClientApplication + ProjectMembership bound to that AccessPolicy via `POST /admin/projects/{id}/client` (Medplum's super-admin atomic-create endpoint; direct `POST /fhir/R4/ProjectMembership` is restricted by Medplum to specific service paths).
5. Performs the OAuth `client_credentials` grant against the new client to obtain a non-admin access token.
6. Asserts four enforcement outcomes against that non-admin token:

| Assertion | Expected | Got | Verdict |
| --- | --- | --- | --- |
| `GET /AuditEvent` (auditor allows AuditEvent read) | 2xx | 200 | PASS |
| `GET /Provenance` (auditor allows Provenance read) | 2xx | 200 | PASS |
| `POST /Observation` (auditor has no clinical write capability) | 4xx | 403 | PASS |
| `GET /Patient` (auditor does not include Patient in resourceRules) | 4xx | 403 | PASS |

The four assertions together prove: (a) the v0.5a generator's emitted AccessPolicy JSON is accepted by Medplum 5.1.8 unmodified, (b) the AccessPolicy is correctly bound to the ProjectMembership via the OSOD-canonical pattern, (c) the binding is enforced at request time — both for resource types named in the policy (positive enforcement: read allowed) and for resource types omitted from the policy (compartment-isolation enforcement: read and write denied).

`npm test` final count: `1063/1063` (baseline v0.4: `1047/1047`; v0.5a foundation commit added +15; this enforcement test adds +1; total +16).

The fixture in this test is auto-provisioned and uses a `Date.now()`-suffixed name on every run. No manual cleanup is required between runs; the artifacts live in the local `medplum` Postgres volume and persist with the rest of the dev project's resources. To prune accumulated fixture artifacts, `docker compose down -v` resets the project entirely.

## Mandate Amendment Candidates

None surfaced during implementation, but two operational lessons were captured (mirrored in `performance-od/.vip/mistakes_log.md`):

1. **"Human-provisioned fixture" is satisfied by env-var credentials, not only by UI clicks.** The original v0.5a build log conflated "Mandate 8 keeps the agent out of auth flows" with "the human must click through the admin UI." The first is binding; the second is an over-conservative interpretation. The agent boundary is about not creating, modifying, or traversing credentials — using credentials a human already provisioned (typically in `osod/.env`) is in-boundary, and it's the established pattern across all live-Medplum integration tests.

2. **Direct `POST /fhir/R4/ProjectMembership` is restricted in Medplum 5.1.8.** Medplum returns "Missing required property" structure errors that suggest a body-shape problem when in fact the endpoint is restricted to specific service paths. The canonical atomic-create path is `POST /admin/projects/{id}/client` (or `/admin/projects/{id}/invite` / `/admin/projects/{id}/bot` for non-client subjects). v0.5b/c/d Codex prompts and any future enforcement-fixture tests should use the admin atomic endpoints rather than direct ProjectMembership FHIR creation.

Both lessons are workflow-process per Mandate 14's "Categories explicitly NOT subject" list — captured for future Tier B recognition-aid candidates if a similar pattern surfaces in v0.5b/c/d.
