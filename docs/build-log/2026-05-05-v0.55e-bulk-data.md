## Decisions

Bulk Data storage stays local-only. `osod_bulk_export_jobs` is the source-of-truth table, and `output_dir` rejects cloud URL schemes.

File download authorization validates OSOD-issued or OSOD-introspected access tokens. Client JWKS validation is kept isolated to client assertion verification at token exchange.

The React consent screen is presentation-only. Backend routes remain authoritative for redirect URI validation, PKCE, OAuth state handling, authorization-code issuance, and audit emission.

CapabilityStatement suppression is severity-aware: mandatory certification claims hard-fail the build when backing tests fail; optional claims may be suppressed.

## Operational Lessons Surfaced During v0.55e Build

The CapabilityStatement rules file needs to live beside its backing tests. Putting capability tests under `mcp/src/__tests__/capability/` lets Pass 4 enforce the claim-to-test map without relying on prose.

File download token validation is clearer when the access-token validator is a separate module from client assertion verification. The separation prevents client JWKS code from drifting into resource-server authorization.

Public conformance documents need application-layer sanitization before WAF caching. Sanitizing only at the proxy layer is too late because the upstream JSON artifact may already contain internal topology strings.

- v0.55e Lesson: ui/package.json build script switched to tsc --noEmit --skipLibCheck due to three-stdlib / @react-three/drei type-resolution timeout. Root cause: deeply nested generic types in @react-three/drei pull-in of three-stdlib shader declarations. Workaround does NOT skip type-checking of OSOD source - only skips node_modules .d.ts files. Forward-gate to v0.6+: deeper fix via tsconfig project references separating ui/src/scenes/* + ui/src/components/{Orbital,PatientOrb}.tsx from the v0.55e SMART UI surface, OR upgrade @react-three/drei + three-stdlib to a version with leaner type declarations.

- v0.55e Lesson: `generateBulkExportJobId()` got retry-on-collision (cap 64 attempts) after laptop-side broad MCP suite verification surfaced 1/1187 flaky failure. Pattern 4 of `PHI_SHAPED_IDENTIFIER_PATTERNS` (`/\b[A-Z][a-z]+[._-][A-Z][a-z]+\b/`) over-matches random base64url output: P ≈ 0.023 per ID across 27 windows × 1000-iteration test loop ⇒ near-certain at-least-one collision per CI run. Codex Cloud's run happened to pass deterministically; Eric's laptop run hit pattern 4. Defense-in-depth fix: keep all four PHI patterns intact (BINDING #25 coverage preserved) and add retry-on-collision in the generator with a 64-attempt cap. Acceptance fixture continues to assert NO PHI-shaped substrings present per BINDING #25; the retry guarantees the constraint can always be met from random bytes. Forward rule: any other defense-in-depth assertion that's checked against random crypto bytes (e.g., file URL paths, polling URL nonces) needs the same generate-and-retry pattern OR explicit narrow-pattern PHI detection. Pass 4 `bulk-data-no-phi-in-job-id` lint scan-surface unchanged — still scans `osod/mcp/src/bulk-data/job-id-generator.ts` for deterministic / patient-derived ID construction.
