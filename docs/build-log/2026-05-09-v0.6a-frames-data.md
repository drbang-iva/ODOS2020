## Decisions

Frames Data population is bulk-file-ingest only in v0.6a. `sync_mode` keeps the forward-compatible enum values, but Frames rows are constrained to `bulk`; `cursor_high_water` remains NULL for Frames sync runs.

The subscription settings surface treats the username as non-secret practice configuration. There is no password field, password secret reference, bearer token, API key, or vendor credential string path.

FHIR catalog identity uses canonical URLs such as `https://osod.dev/catalog/frames/<sku>`. Provenance targets reference FHIR canonical URLs or the ingest Task, not raw SQL row primary keys.

The SQL practice inventory table stores the practice identifier and enforces tenant isolation through RLS. A true relational FK is deferred until OSOD has a first-class practice SQL table; the current Medplum-backed practice identity does not give this migration a stable local FK target without adding a sixth v0.6a table.

## Mandate 15 Boundary-Calibration Audit

Actor-axis check: v0.6a keeps the operator/upload actor boundary intact. Catalog population begins with a local operator-uploaded file; the Practice Settings panel stores only non-secret username/config state and an active-subscription toggle. No agent path collects a Frames Data password, API key, bearer token, or portal credential.

Existing-pattern check: the slice composes with existing OSOD patterns instead of adding new infrastructure primitives. Audit events reuse `osod_audit_events` plus FHIR AuditEvent projection; provenance uses canonical FHIR targets; UI reads route through existing FHIR helpers; privileged CSV export uses the existing presentation-role model while SMART `DeviceDefinition` access is hardened in the existing scope-intersection layer.

External-precondition check: the only external precondition is a human/operator having a lawful Frames Data subscription file to upload locally. OSOD does not discover Frames Data endpoints, call framesdata.com, scrape a portal, or redistribute raw catalog rows through public APIs. HCPCS/SNOMED validation routes through local terminology tables/dictionaries, not external terminology services.
