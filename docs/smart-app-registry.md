# OSOD SMART App Registry

v0.55b adds a local SMART app registry inside the practice-owned Docker Compose stack. The canonical app record is a FHIR R4 Endpoint or Device with the OSOD `smart-client-app` extension. Medplum's local app-client record is generated by the adapter in `data/medplum-adapters/` only.

## Registration

Apps register through:

```text
POST /oauth2/register
```

The endpoint follows RFC 7591 client metadata behavior and returns `201 Created` with `client_id`, optional `client_secret`, and the registered metadata. OSOD also requires the local seed-manifest fields at registration:

```json
{
  "client_name": "Local SMART App",
  "redirect_uris": ["http://127.0.0.1:5173/callback"],
  "token_endpoint_auth_method": "none",
  "scope": "user/Observation.rs",
  "scope_request_canonical": "user/Observation.rs",
  "risk_class": "low",
  "phi_boundary": "metadata-only",
  "launch_mode": "ehr",
  "network_egress": "local-only",
  "external_services_required": false,
  "baa_required": false,
  "image_analysis_prohibited": true,
  "allowedJurisdictions": ["US"],
  "prohibitedStates": []
}
```

## Canonical Shape

The canonical record is an Endpoint by default. A request can set `app_shape` to `device` when the app is better represented as a managed local software component. Both shapes carry:

```text
https://osod.dev/fhir/StructureDefinition/smart-client-app
```

The extension carries OAuth metadata plus risk, PHI boundary, launch mode, network egress, BAA, image-analysis, jurisdiction, and canonical-scope fields. OSOD application code reads this canonical record and does not read the Medplum adapter record directly.

## Admin Review

Practice admins review seed-manifest fields before installation. Install validation blocks:

- `phi_boundary = patient-payload` when `baa_required` is not true.
- `image_analysis_prohibited = false`.
- Practice jurisdiction conflicts with `prohibitedStates`.
- Practice jurisdiction missing from a non-empty `allowedJurisdictions` list.

Compatibility gaps between app-required SMART capabilities and OSOD-supported capabilities are shown as warnings. Installation proceeds only with admin attestation.

## Geographic Fencing

South Carolina autonomous-refraction installs are hard-blocked when the app declares a prohibited `US-SC` jurisdiction. The citation pinned at v0.55b session open is the South Carolina Supreme Court opinion issued January 21, 2026, paired with South Carolina Code Chapter 24:

- <https://www.sccourts.org/media/opinions/HTMLFiles/SC/28310.pdf>
- <https://www.scstatehouse.gov/code/t40c024.php>

## BAA Attestation

Apps that carry patient payloads must require a BAA. v0.55b records that as metadata and blocks installs when the invariant is false. OSOD still runs local-only; this is an app-review safety rule, not a cloud-hosting path.

## Image Analysis

Mandate 13 remains unchanged. Seed-manifest apps default to `image_analysis_prohibited: true`, and v0.55b ships no app with that value set to false. The field is enforced at both the StructureDefinition and install-validation layers.

## Device Resource: App vs Agent

Endpoint is the default shape for launchable SMART apps. Device is available when the practice needs to manage a local software component as an operational device. Device does not mean autonomous clinical agency; app install still requires human-supervised practice-admin review.

## Why This Is Not A Marketplace

The v0.55 ledger row 25 keeps the operational claim PROVISIONAL under Mandate 14-amendment 2026-05-01b. v0.55b ships infrastructure and a static local eyecare SMART app seed catalog only. No remote app distribution, app-store API, PerformanceOD-hosted sync, or cloud catalog is part of this slice.
