# v0.4 Mandate 4 Infrastructure Audit

Date: 2026-04-28
Baseline reviewed: post-v0.35 close at `c06d89d`, current repo head `715afc1`
Scope: v0.4a foundation only. Clinical dry-eye, ortho-K, and myopia-management workflows remain out of scope for this slice.

## 1. Timeline Event System

Status: partial

Current state: v0.35 stores longitudinal FHIR resources and writes Provenance for clinical write tools, but event types are implicit in individual tool implementations. There is no declarative module hook for future clinical modules to register event types or timeline surfaces.

Gap filled in v0.4a: add module manifests under `data/modules/` with event-type declarations and a tool/profile registration surface. v0.4a write tools create mandatory Provenance sidecars so timeline construction has a durable source-of-change substrate.

Acceptance criterion: `data/modules/contact-lens-foundation.yaml` declares the contact-lens foundation event types and MCP tool surfaces; v0.4a write-tool tests verify Provenance is default-on.

## 2. Equipment Registry

Status: partial

Current state: v0.35 can attach measuring instruments to Observations through native FHIR references, but there is no configured practice registry of fitting devices or manufacturer catalog blueprints.

Gap filled in v0.4a: seed a FHIR `DeviceDefinition` registry for contact-lens lab/catalog blueprints and add a type-aware `create_device_definition` MCP tool. Patient-specific contact lenses reference their catalog blueprint through `Device.definition`.

Acceptance criterion: the profile installer idempotently installs at least five `DeviceDefinition` seed resources and tests retrieve them by OSOD identifier.

## 3. Medication/Product Timeline

Status: partial

Current state: v0.35 has longitudinal clinical write surfaces and Provenance, but product/device/material changes are not unified.

Gap filled in v0.4a: represent contact lenses as patient-specific `Device` resources, materials/coatings as `Substance`, lens geometry as `Device.property[]`, and every create/update write as Provenance-backed. The same pattern can later include medication and material episode changes without a schema fork.

Acceptance criterion: `create_lens_device`, `update_lens_device_properties`, and `create_substance` ship with mandatory Provenance and `X-OSOD-Source` headers.

## 4. Document/Photo Management

Status: shipped

Current state: v0.35 raw asset preservation already creates categorized `DocumentReference` entries with SHA-1/FHIR attachment support and OSOD SHA-256 preservation; those resources can be linked from Observations through standard `Observation.derivedFrom` when a measurement is computed from raw source material.

Gap filled in v0.4a: no new asset store is introduced. v0.4a codifies cross-resource linkage discipline: `Observation.derivedFrom` may target raw source resources, while lens-fit Observations use `Observation.focus = Device/<lens-id>`.

Acceptance criterion: schema-prophylaxis tests ensure v0.4a code does not point `Observation.derivedFrom` at a contact-lens `Device`.

## 5. Module Registration System

Status: not-built

Current state: profiles and terminology are installed from static directories, but modules do not declare their routes, event types, profiles, terminology, or MCP tools.

Gap filled in v0.4a: introduce declarative YAML manifests at `data/modules/core.yaml` and `data/modules/contact-lens-foundation.yaml`. These manifests are intentionally data-only in v0.4a; routing/UI activation waits for later module slices.

Acceptance criterion: the contact-lens foundation manifest declares v0.4a profiles, CodeSystems, ValueSets, event types, and the ten MCP tool surfaces added in this slice.

## 6. Fitting/Trial/Remake Tracking

Status: not-built

Current state: no shared fitting/trial/remake representation exists in v0.35.

Gap filled in v0.4a: create OSOD terminology for contact-lens fitting events and fit findings, use `Device` for the lens being evaluated, use `Observation.focus` for lens-fit findings, and add `get_lens_fit_history` as the longitudinal query surface. Workflow screens remain v0.4c.

Acceptance criterion: contact-lens builder tests prove fit findings are patient-subject Observations focused on the lens `Device`, and query tests retrieve lens-fit history through the standard focus search path or documented fallback.

## 7. Device Data Import Pipeline

Status: partial

Current state: v0.35 can store raw device exports as `DocumentReference`, but structured import ingress is not separated from later DICOM/PACS work.

Gap filled in v0.4a: add ocular measurement graph query helpers and Observation profiles for K readings, pachymetry, meibomian gland scoring, TBUT, Schirmer, and contact-lens fit findings. Device-generated values use `Observation.device`; computed values use `Observation.derivedFrom` to raw source resources only. DICOM/PACS gateway remains deferred to v0.6.

Acceptance criterion: v0.4a query tools can retrieve Observation histories by patient, code, optional laterality, optional date range, and Device focus; search conformance is documented with the fallback path when `Observation?focus=...` is unavailable.
